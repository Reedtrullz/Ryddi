# Ryddi v0.4 Guided Cleanup and E2E Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` to implement this plan task by task. Use `superpowers:test-driven-development` for each behavior change and `superpowers:verification-before-completion` before each commit or release claim.

**Goal:** Turn Ryddi from a collection of storage reports into a coherent, confidence-building cleanup workflow, while preserving the fail-closed safety contracts established in `v0.3.1` and proving the complete user journey across supported window sizes.

**Architecture:** Keep `ReclaimerCore` as the only authority for classification, selection eligibility, plans, execution, and receipts. Add a typed `CleanupJourneySnapshot` that translates that authority into one next decision for the app. Replace the 25-destination dashboard with five task-oriented destinations, then split the app model into focused stores. Use native macOS `List`, `Table`, `NavigationSplitView`, toolbar, sheet, and inspector patterns. Remote Targets remain read-only and move to one bounded, allowlisted SSH batch per evidence phase.

**Tech Stack:** Swift 6, SwiftUI, SwiftPM, macOS 14+, XCTest, Accessibility APIs, `/usr/bin/ssh`, existing Ryddi packaging and notarization scripts.

**Depends on:** `docs/superpowers/plans/2026-07-14-ryddi-v0.3.1-correctness-and-state.md` is complete and released. In particular, this plan consumes `DashboardActivity`, `ScanControl`, `ExecutionReconciliation`, `AuditStoreSnapshot`, operational permission probes, and deterministic known-host evidence.

**Release target:** `v0.4.0`, bundle version `5`.

## Non-Negotiable Safety Contracts

- Never infer cleanup permission from a screen, row, queue, or owner label. Only `ReclaimerCore` can mark an item eligible.
- Keep one-use, identity-bound Trash authorization and final metadata/classification/open-handle checks.
- Protected data, browser profiles, VM/container disks and volumes, Codex sessions/memories/config/auth, credentials, databases, GarageBand/Logic assets, and unknown user data remain unselected by default.
- Remote Targets remain report-only. Do not add remote execute, prune, reset, sudo-password, key-storage, or unattended action paths.
- Do not add telemetry, cloud analysis, path upload, root helpers, or Mac App Store sandboxing.
- The UI may simplify language and order, but must not weaken safety classes, typed conditions, non-claims, or receipt evidence.
- Before long build/test loops, run `df -h /System/Volumes/Data` and stop below `30Gi` free.
- Use `swift test --scratch-path "$PWD/.build"` and `swift build --scratch-path "$PWD/.build"`.

```swift
public enum CleanupJourneyStep: String, Codable, Hashable, Sendable {
    case needsAccess
    case needsScan
    case recommendations
    case planReview
    case dryRunReady
    case confirmationReady
    case completed
    case verificationRequired
}

public enum CleanupJourneyAction: String, Codable, Hashable, Sendable {
    case reviewAccess
    case scan
    case reviewRecommendations
    case buildPlan
    case runDryRun
    case authorizeTrash
    case executeTrash
    case verifyCleanup
    case viewReceipt
}

public struct CleanupRecommendationGroup: Identifiable, Hashable, Sendable {
    public let id: String
    public let title: String
    public let summary: String
    public let findingIDs: [String]
    public let allocatedBytes: Int64
    public let estimatedReclaimBytes: Int64
    public let nextAction: ReviewNextAction
    public let selectable: Bool
}

public struct CleanupJourneySnapshot: Hashable, Sendable {
    public let step: CleanupJourneyStep
    public let title: String
    public let summary: String
    public let primaryAction: CleanupJourneyAction
    public let secondaryActions: [CleanupJourneyAction]
    public let groups: [CleanupRecommendationGroup]
    public let selectedFindingIDs: Set<String>
    public let expectedReclaimBytes: Int64
    public let receiptSummary: String?
    public let nonClaims: [String]
}

public struct CleanupJourneyInput: Sendable {
    public let permissionReport: PermissionAdvisorReport
    public let scanSession: ScanSession?
    public let findings: [Finding]
    public let plan: ReclaimPlan?
    public let latestReceipt: ExecutionReceipt?
    public let reconciliation: ExecutionReconciliation?
    public let selectedFindingIDs: Set<String>
}

public enum CleanupJourneyBuilder {
    public static func build(from input: CleanupJourneyInput) -> CleanupJourneySnapshot
}
```

### Step 1: Write the state-machine tests first

Cover this precedence exactly:

1. A completed execution awaiting rescan is `.verificationRequired` with `.verifyCleanup` primary.
2. A clean verified receipt is `.completed` with `.viewReceipt` primary.
3. A confirmed clean dry-run is `.confirmationReady`.
4. An existing plan without a matching clean dry-run is `.dryRunReady`.
5. Eligible selected recommendations are `.planReview`.
6. Current findings without a selection are `.recommendations`.
7. No current scan is `.needsScan` unless configured scan roots are operationally blocked.
8. Operationally blocked essential roots are `.needsAccess`.

Also prove:
- Exactly one primary action exists for every state.
- Protected and `neverTouch` findings are never selectable.
- `reviewRequired`, `preserveByDefault`, and native-tool findings never contribute to Trash reclaim.
- A stale plan or receipt from another `ScanSession.id` is ignored.
- Empty results produce a useful completed/no-waste state, not a disabled page.

Run:

```bash

```

Expected: fail because the builder does not exist.

### Step 2: Implement a pure deterministic builder

- Do not read disk, audit files, or global app state in the builder.
- Group by user intent: Safe Maintenance, Quit App First, Native Tool, Valuable History, Personal/App Assets, Unknown.
- Use existing safety and `ReviewNextAction` values; do not derive eligibility from English copy.
- Carry explicit non-claims for estimated reclaim and protected/review-only content.

### Step 3: Prove compatibility with current planning logic

For a mixed finding fixture, assert the set selected by `CleanupJourneyBuilder` equals the set accepted by `PlanBuilder`; any mismatch must fail closed and leave the finding unselected.

```swift
public enum RemoteReadOnlyCommandID: String, CaseIterable, Codable, Sendable {
    case uname
    case hostname
    case username
    case home
    case osRelease
    case diskBlocks
    case diskInodes
    case availableTools
    case sudoCapability
    case directoryUsage
    case largeFiles
    case journalUsage
    case aptCache
    case dockerSystemUsage
    case dockerObjects
}

public struct RemoteReadOnlyBatch: Sendable {
    public let nonce: String
    public let commandIDs: [RemoteReadOnlyCommandID]
    public func shellProgram() throws -> String
}

public enum RemoteBatchParser {
    public static func parse(
        stdout: String,
        stderr: String,
        nonce: String,
        expected: [RemoteReadOnlyCommandID]
    ) throws -> [RemoteCommandResult]
}
```

### Step 1: Write safety and parser tests first

Prove:
- Probe performs one fake SSH invocation; scan performs one fake SSH invocation.
- Only static allowlisted command fragments can enter a batch.
- The nonce is random hexadecimal and cannot come from target/output input.
- Missing, duplicated, reordered, oversized, or malformed section markers fail closed.
- Per-command exit status, bounded stdout/stderr preview, and command ID remain available in receipts.
- Every SSH invocation retains `BatchMode=yes`, zero password prompts, strict host-key checking, timeout/output bounds, and no agent forwarding.
- No generated batch contains delete, prune, reset, package mutation, Docker mutation, or sudo cleanup.

### Step 2: Implement a static shell envelope

Build the program only from enum-owned literals. Wrap each command with nonce-qualified begin/end/status markers and `LC_ALL=C`. Do not interpolate remote paths from user input; scanned roots remain the existing fixed allowlist.

### Step 3: Preserve degraded evidence

An unavailable tool or permission-denied command becomes an individual failed `RemoteCommandResult`; it must not discard successful filesystem evidence. Structural marker failure invalidates the whole batch.

### Step 4: Wire probe and scan

- Probe batch: identity, OS, home, filesystems, tools, non-interactive sudo capability.
- Scan batch: directory usage, large files, journald, apt cache, and Docker inventory.
- Continue producing the existing `RemoteProbeReport` and `RemoteScanReport` shapes.
- Keep `remote native` and `remote plan` as guidance/report paths only.
